    Introduction

    SOLID is a mnemonic for the 1st 5 principles of class design as codified by "Uncle Bob", a.k.a. Robert C. Martin. These principles are 'ancient' by software engineering standards, but they are (please forgive me) a SOLID set of design guidelines that are easy ways to improve the quality of the design. Even though the blog-o-sphere is already inundated with various explanations of SOLID, I still felt I have to try my own hand at explaining these principles.
    SRP Single Responsibility Principle A class should have one, and only one, reason to change Config
    OCP Open-Closed Principle You should be able to extend a classes behavior, without modifying it Config
    LSP Liskov Substitution Principle Derived classes must be substitutable for their base classes Java Collection Library
    ISP Interface Segregation Principle Make fine grained interfaces that are client specific Messaging/DAO
    DIP Dependency Inversion Principle Depend on abstractions, not on concretions Messaging - static method

    My most recent position required extending a pre-existing code base. After delving into this code base, I was shocked by the number of commonplace violations of these principles and I wondered why they weren't being followed. One of my concerns was a lack of awareness or at least a lack of easy understanding of these principles. Which is really why I wanted to write a blog post about SOLID.

    One convenient thing was that there was one software component would violated the first 2 principles of SOLID, so this is a good example for describing those SOLID principles.

    So this code base was a LARGE - used to build different Java programs (run as Unix daemons) deployed to different types of nodes. The code used a simple configuration system that wouldn't be hard to imagine what this looked like.

    public class Config {
    public static Config getInstance() { ... }
    private Config() { ... }
    public String getMessageBrokerUri() { ... }
    public String getDatabaseUri() { ... }
    public String getClientCertificatePath()) { ... }
}

    Of course, this configuration class grew as the codebase grew, adding dozens and dozens of 'getXXX()' property methods. If the above interface grates on anyone, we did end up replacing it with a better system later, but that's a discussion for future post.

    SRP: Single-Responsibility Principle

    A class should have only one reason to change.

    However, I always felt this description was difficult to understand, but looking at examples help a lot. The classic example is a report of some kind: a report has 'items' to report about and it has ways to output the report - 2 relatively separate concerns. If you can forecast reasonable ways a module may need to change, then, for 2 completely different kinds of forecasted changes, you can check if the code affected by those changes are intermingled or not.

    Our example Config system absolutely 'jumps' out as a violation of 'SRP' once you recognize the pattern. This class clearly has 2 'sides' to it * a way for production code to retrieve the configuration values it needs (regardless of where and how they are stored), and * a way for reading the storage mechanism in which configuration are held, ideally independently of the specific properties and property types kept in it.

    In classic SRP parlance, this class had two reasons to change * it may need to support new configuration properties and configuration property types * it may need to support different mechanisms of storage of those configuration definitions

    So, this class need to be split in two: * an interface for accessing properties, and * one or more implementations for reading properties from the storage medium.

    This split helps the design because unit tests would now have an interface to mock or create a stub from so that they aren't forced to rely on production behavior. Meanwhile, the production implementation(s) can perform more complex behaviors like reading a hierarchical set of configuration files or read the configuration from a database or other complex store that uses authentication and/or encryption. Finally, the config system becomes more reusable because library modules can use interface with a dependency on a specific implmenetation - those libraries are now usable by different main programs that may want to use different storage mechanisms for configuration properties.

    public interface Config {
    public static final String MESSAGE_BROKER_URI_KEY = ...;
    public static final String DATABASE_URI_KEY = ...;
    public static final String CLIENT_CERT_PATH_KEY = ...;
    public String getProperty(String key) { ... }
}

    public class ConfigFiles implements Config {
    public static getInstance() { ... }
    private ConfigFiles() { ... }
    public String getProperty(String key) { ... }
}

    Open-Closed Principle

    Software entities should be OPEN for extension, but CLOSED for modification

    So ... what the eff does this mean? More simply put (I hope), a software component that is intended to be extended (in support of growing requirements/codebase), then should be extensible in a way that does NOT require modifying the original code each time.

    So how does our configuration system violate the Open-Closed Principle? Well, it had methods like "String getMessageBrokerUri()" and "String getDatabaseUri()". As we added more features and functionality to the code, we would inevitably add new methods 'getXXX()' to support configration for new features. This component violated the open-closed principle because each new feature added required modifying the original core configuration component.

    To comply with the open-closed principle, we should have designed an application property system that allowed new features to access new configuration properties without having to modify the original configuration code.

    The Liskov Substition Principle

    Derived classes must be substitutable for their base classes

    Aren't derived classes always substitutable for their base classes? Syntactically, yes, but this principle isn't about syntax - it's largely about behavior. To understand this principle, we need only look at the Java Collection library for numerous examples.
